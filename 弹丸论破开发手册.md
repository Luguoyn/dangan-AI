## 零、手册使用指南

### 0.1 手册定位

本手册是给 **OpenCode AI Agent** 看的 "建筑图纸"。OpenCode 将按照本手册的规划，**一步一步**用 Godot4 搭建出一个完整的弹丸论破风格游戏框架。

**关键理念**：
- 游戏引擎（Godot）是 **解释器**，不是写死剧情的容器
- 剧本文件是 **数据**，人类编剧用 JSON/Tres 编写，游戏引擎读取并演出
- OpenCode 的每一步工作，都是搭建这个 "解释器" 的某个模块

### 0.2 术语速查表

| 术语 | 通俗解释 | 首次出现章节 |
|------|----------|-------------|
| **学级裁判** | 找出真凶的辩论环节，类似法庭辩论 + 小游戏集合 | 一 |
| **言弹** | 证据/证词做成的 "子弹"，用于击破矛盾发言 | 一 |
| **无休止议论** | 多名角色循环发言，玩家需要找到矛盾点并发射言弹 | 二.6 |
| **反论** | 单一角色全力反驳主角，需要快速"斩断"对方言辞 | 二.7 |
| **拼字** | 在旋转/飘动的字符（字母、偏旁、汉字）中按顺序选出正确的，拼出关键词（英文单词、汉字、汉语词汇） | 二.8 |
| **高潮再现推理** | 用漫画分格方式还原整个案件真相 | 二.9 |
| **黑白熊** | 游戏的吉祥物/管理者，宣布规则、处刑 | 三 |
| **精神力** | 玩家的"生命值"，受伤害会减少，归零则裁判失败 | 二.4 |

## 一、游戏基本设定与篇章循环

### 1.1 核心架构：三篇章循环

```
┌──────────┐    发生事件    ┌──────────┐    调查完成    ┌──────────┐
│  日常篇   │ ──────────→ │  非日常篇  │ ──────────→ │  学级裁判  │
│ (2D场景)  │              │ (2D场景)   │              │ (3D场景)   │
└──────────┘              └──────────┘              └──────────┘
     ↑                                                    │
     │                 裁判结束，处刑/存活                  │
     └────────────────────────────────────────────────────┘
```

#### 1.1.1 日常篇（Daily Life）

**玩法描述**：
- 玩家控制主角在 2D 俯视角/侧视角场景中自由移动
- 与场景中的角色对话，增进好感度
- 触发固定剧情事件
- 可收集隐藏要素（如黑白熊金币）

**状态机**：

```
[场景加载] → [自由移动] ←→ [菜单/存档]
                 ↓ 触发事件
              [对话事件] → [返回自由移动]
                 ↓ 所有事件完成
              [进入非日常篇]
```

**简单实现思路**：
- 使用 `TileMap` 或 `Sprite2D` 构建 2D 场景
- 角色用 `CharacterBody2D` 或简单的 `Sprite2D` + 点击检测
- 对话系统由剧本驱动（详见第四章）
- 场景切换通过 `SceneManager` 单例控制

#### 1.1.2 非日常篇（Deadly Life）

**玩法描述**：
- 发生杀人事件后，主角查看尸体、调查现场
- 收集 **言弹**（证据与证词）
- 与角色对话获取证词
- 调查完成后强制进入学级裁判

**言弹系统**：
- 每个言弹是一个数据对象：`{id, name, description, type, icon}`
- 言弹存储在一个全局数组（`EvidenceManager`）中
- 调查点和对话会触发 `add_evidence` 指令

**简单实现思路**：
- 复用日常篇的 2D 场景框架
- 增加 "调查模式" —— 点击特定区域弹出调查 UI
- 言弹用 `Resource` 类型管理（`EvidenceItem.gd`）
- GUI 上显示已收集言弹列表（类似背包界面）

#### 1.1.3 学级裁判篇（Class Trial）

**全部在 3D 场景中进行**，是游戏的核心和高潮。详见第二章。

### 1.2 场景与角色数量规划

- **角色总数**：16 名 "超高校级" 高中生 + 黑白熊（管理者）= 17 个实体
- **场景数量**：每个篇章约 4-8 个场景（教室、走廊、体育馆、宿舍等）
- **裁判场**：一个中心 3D 场景（圆形辩论场），16 个站位围成圈

### 1.3 引擎架构总览

```
GameManager (Autoload 单例)
├── EventBus              ← 全局信号总线
├── ScriptInterpreter     ← 剧本解释器（核心）
├── SceneManager          ← 场景加载/切换
├── EvidenceManager       ← 言弹数据管理
├── CharacterManager      ← 角色数据管理
├── AudioManager          ← 音频播放
├── SaveLoadManager       ← 存档/读档
├── DebateManager         ← 辩论总控（HP、难度、小游戏调度）
├── HangmanManager        ← 拼字统一调度器（三种模式）
├── DifficultyConfig      ← 难度配置（Resource）
└── LocaleManager         ← 多语言文本管理
```

## 二、学级裁判详细机制（3D）

> 学级裁判是整个游戏最复杂的部分。以下按 **出现顺序** 逐一讲解每个小游戏的机制和实现思路。

### 2.0 裁判场节点结构（3D 场景蓝图）

```
ClassTrialScene (Node3D)
├── Camera3D                   ← 主摄像机
├── CameraAnimator (AnimationPlayer) ← 摄像机运镜
├── Environment (WorldEnvironment)  ← 灯光/雾/后处理
├── CircleStage (Node3D)
│   ├── Podium01-16 (Node3D)   ← 16个站位（含角色模型位置）
│   │   ├── CharacterModel     ← 3D角色模型（或立绘板）
│   │   └── NameTag (Label3D)  ← 角色名字标签
│   └── CenterCircle (MeshInstance3D) ← 中心装饰
├── UI (CanvasLayer)
│   ├── DialogueBox            ← 对话框面板
│   ├── CharacterNameTag       ← 说话者名字
│   ├── SkillSelectPanel       ← 技能选择面板
│   ├── EvidenceListPanel      ← 言弹列表
│   ├── HPBar                  ← 精神力血条
│   ├── TimerLabel             ← 倒计时
│   └── MiniGameContainers     ← 各小游戏UI容器
│       ├── NonStopDebateUI
│       ├── RebuttalUI
│       ├── HangmanUI
│       └── ClimaxInferenceUI
├── DebateManager (Node)       ← 辩论流程控制（核心脚本）
└── ScriptRunner (Node)        ← 剧本指令执行器
```

> **关键约定：config_id 到 .tres 的映射规则**  
> 剧本中的 `config_id`（如 `"debate_ch01_01"`）对应资源路径 `res://resources/debate_configs/debate_ch01_01.tres`。  
> `DebateManager` 加载配置时统一使用此路径规则：`"res://resources/debate_configs/" + config_id + ".tres"`。  
> 同理，拼字配置在 `res://resources/debate_configs/hangman/`、反论配置在 `res://resources/debate_configs/rebuttal/`、高潮再现配置在 `res://resources/debate_configs/climax/`。  
> **所有小游戏配置都是 Godot Resource（.tres）文件**，可在 Godot 编辑器的 Inspector 面板中直接编辑，无需手写 JSON。

### 2.1 开局准备

#### 2.1.1 技能选择（Skill Select）

**玩法描述**：
- 裁判开始前，玩家从已习得技能中选择 1-2 个装备
- 技能可提供：降低小游戏难度、增加分数倍率、自动瞄准等辅助效果
- 技能消耗 "SP"（技能点数），有携带上限

**技能数据结构**：
```gdscript
# SkillItem.gd (Resource)
class_name SkillItem extends Resource
@export var skill_id: String
@export var skill_name: String        # e.g. "美声"
@export var description: String       # e.g. "无休止议论中准星范围扩大30%"
@export var sp_cost: int = 1
@export var effect_type: String       # "aim_assist" / "time_extend" / "damage_reduce"
@export var effect_value: float = 0.3
```

**实现思路**：
- 在裁判场UI弹出 `SkillSelectPanel`
- 遍历玩家已解锁的 `SkillItem` 列表，显示为可选按钮
- 选择后存入 `DebateManager.equipped_skills` 数组
- 在小游戏中，对应的管理器检查是否有匹配的技能并应用效果

#### 2.1.2 言弹浏览（Evidence Review）

**玩法描述**：
- 玩家可以浏览所有已收集的言弹
- 点击言弹查看详细信息（名称、描述、图标）
- 确认准备完毕，点击"开始裁判"

**实现思路**：
- `EvidenceListPanel` 是一个 `ScrollContainer` + `VBoxContainer`
- 每个言弹显示为 `EvidenceItemButton`（图标+名称）
- 点击弹出 `EvidenceDetailPopup`（描述文字）
- "开始"按钮触发 `DebateManager.start_debate()` 信号

### 2.2 剧本对话推进（Story Dialogue）

**玩法描述**：
- 裁判中穿插大量剧本对话
- 角色围在裁判场的圆形站台上轮流发言
- 说话时摄像机自动切到发言人特写
- 台词逐字显示，玩家点击推进

**摄像机运镜逻辑**：

```
角色A发言 → Camera从当前位置 → 飞到角色A前方特写位置 → 摄像机震动/呼吸效果
角色B发言 → Camera从A位置 → 飞到角色B前方特写位置 → ...
全员讨论 → Camera拉远，俯视全场
```

**实现思路**：

```gdscript

    tween.tween_property(camera, "position", target + camera_offset, 0.8).set_ease(Tween.EASE_IN_OUT)
    tween.parallel().tween_property(camera, "look_at", target, 0.8)

func shake_camera(intensity: float = 0.5):
    # 给摄像机加微小随机偏移
    pass
```

**点击交互逻辑**：

```gdscript
# DialogueBox.gd
var full_text: String = ""
var displayed_text: String = ""
var char_index: int = 0
var is_text_complete: bool = false

func _input(event):
    if event is InputEventMouseButton and event.pressed:
        if not is_text_complete:
            # 当前句子没显示完 → 立即显示全部文字
            displayed_text = full_text
            char_index = full_text.length()
            is_text_complete = true
            update_label()
        else:
            # 当前句子已显示完 → 发送信号进入下一句
            EventBus.emit_signal("dialogue_next")

func typewrite_effect():
    # 每帧显示一个字符的逐字效果
    while char_index < full_text.length():
        displayed_text += full_text[char_index]
        char_index += 1
        update_label()
        await get_tree().create_timer(0.05).timeout
    is_text_complete = true
```

**自动/手动模式切换**：
- GUI上放置一个 `AutoButton`
- 手动模式：玩家点击推进
- 自动模式：每句显示完后等待固定时间（如 2 秒）自动进入下一句
- 使用 `Timer` 节点实现

### 2.3 对话分支选项（Dialogue Choices）

**玩法描述**：
- 在特定对话中弹出 2-4 个选项
- 玩家选择后影响：
  - 特定角色的 **好感度**（影响角色结局和技能获取）
  - 小段对话分支（选错可能被纠正，但不影响主线真相）
  - 精神力增减

**数据结构**：
```json
{
  "type": "choice",
  "prompt": "你认为真凶是谁？",
  "choices": [
    {"text": "雾切同学", "next": "choice_kirigiri", "affection": {"kirigiri": 5}},
    {"text": "十神同学", "next": "choice_togami", "affection": {"togami": -3}}
  ],
  "correct_index": 0,
  "wrong_penalty": 10  // 精神力扣除
}
```

**实现思路**：
- `ChoicePanel` 弹出，创建 `Button` 列表
- 选择后触发 `EventBus.choice_made(index)`
- 剧本解释器根据选择跳转到对应剧本段
- `AffectionManager` 更新好感度数值

### 2.4 精神力系统（HP System）

**贯穿整个学级裁判**，是玩家的"生命值"。

**玩法描述**：
- 精神力初始为 100（或根据难度调整）
- 在小游戏中犯错（选错言弹、拼错字、反论失误）会扣减
- 归零 → 裁判失败 → 全员处刑 Bad End
- 成功击破矛盾可以恢复少量精神力

**实现思路**：
```gdscript
# DebateManager.gd
signal hp_changed(current_hp, max_hp)
signal hp_depleted()

var max_hp: int = 100
var current_hp: int = 100

func damage_hp(amount: int):
    current_hp = max(0, current_hp - amount)
    hp_changed.emit(current_hp, max_hp)
    if current_hp <= 0:
        hp_depleted.emit()

func heal_hp(amount: int):
    current_hp = min(max_hp, current_hp + amount)
    hp_changed.emit(current_hp, max_hp)
```

#### 2.4.1 难度系统（Difficulty Scaling）

难度影响所有小游戏的参数。建议定义三种难度：

```gdscript
# DifficultyConfig.gd (Resource)
class_name DifficultyConfig extends Resource
@export var difficulty_name: String          # "kind" / "normal" / "mean"
@export var evidence_ring_time: float        # 言弹选择环的展开速度（秒）
@export var nonstop_yellow_linger: float     # 黄色文字在屏幕上的停留时间（秒）
@export var nonstop_noise_density: float     # 杂音密度倍率
@export var hangman_time_bonus: float        # 拼字额外时间（秒）
@export var hangman_rotation_speed: float    # 拼字环旋转速度倍率
@export var rebuttal_judge_window: float     # 反论判定容差（像素）
@export var rebuttal_speed_mult: float       # 反论语句速度倍率
@export var climax_time_limit: float         # 高潮再现时间限制（秒）
@export var hp_damage_mult: float            # 所有伤害倍率
@export var hp_heal_amount: int              # 成功击破时回复量
@export var max_hp: int                      # 最大精神力
```

三个预置值示例：

| 参数 | 温情(Kind) | 普通(Normal) | 残酷(Mean) |
|------|-----------|-------------|-----------|
| 无休止议论·黄字停留 | 12秒 | 8秒 | 5秒 |
| 拼字时间奖励 | +10秒 | +0秒 | -5秒 |
| 反论判定容差 | 80px | 50px | 30px |
| 伤害倍率 | 0.7× | 1.0× | 1.5× |
| 最大HP | 120 | 100 | 80 |

`DebateManager` 在裁判开始时加载 `DifficultyConfig`，所有小游戏管理器读取其参数来调整自身行为。

### 2.6 无休止议论（Nonstop Debate）

> **首次出现位置**：每次学级裁判中，当讨论进入白热化阶段，多名角色开始循环发表观点时触发。

#### 2.6.1 核心玩法

1. **文字飘动**：多名角色的发言文字以不同方向、不同速度在屏幕上飘动
2. **寻找矛盾**：其中一句 **黄色文字** 与其他角色的证言或已知证据矛盾
3. **瞄准与发射**：
   - 玩家移动准星（`Control` 节点跟随鼠标/手柄摇杆）
   - 对准黄色矛盾语句
   - 按确认键打开言弹选择环（或快速选择）
   - 选择正确的言弹，发射击破
4. **杂音干扰**：浅色/灰色文字是杂音（无关发言），瞄准经过时会轻微干扰准星
5. **长按蓄力**：部分言弹需要长按蓄力才能击破（增加紧张感）

#### 2.6.2 详细机制

**文字飘动规则**：
- 每句发言是一个 `DebatePhrase` 对象
- 飘动方向：随机角度（0°~360°），从屏幕中心向外辐射或随机方向
- 飘动速度：非黄色文字较快（容易飘出屏幕），黄色文字较慢（给玩家反应时间）
- 屏幕边缘反弹或循环出现

**言弹选择方式**（推荐两种模式可选）：
- **环形菜单**：按住 Tab 键，已收集的言弹以环形排列在准星周围，摇杆选择
- **列表选择**：按 Tab 弹出列表，上下键选择

**发射与命中的反馈**：
- 正确言弹命中黄色文字 → 子弹飞行动画 + 破碎特效 + 主角台词 "就是这个！"（コレダ！）
- 错误言弹命中 → 子弹弹开特效 + 精神力扣减 + 该黄色文字短暂保护（无法再次瞄准）
- 未命中 → 子弹飞出屏幕，不扣精神力（但浪费了言弹）

**多目标无休止议论**：
- 一场议论中可以有多条黄色文字（多条矛盾），需要依次击破
- 击破第一条后，其他讨论继续，残留的黄色文字仍在飘动
- 全部击破后议论结束
- `NonStopDebateConfig` 中新增字段 `require_sequential: bool`：
  - `true`（默认）：必须按指定顺序击破，击错顺序扣HP
  - `false`：可以先击破任一黄色文字，提升自由度

#### 2.6.3 数据结构

```gdscript
# DebatePhrase — 单条辩论发言
class_name DebatePhrase extends Resource
@export var text: String                    # "被害者是在图书馆被发现的..."
@export var speaker_id: String             # "asahina"
@export var is_contradiction: bool = false  # 是否为矛盾黄色语句
@export var required_evidence_id: String    # 击破此矛盾需要的言弹ID（仅黄色语句有值）
@export var speed: float = 30.0            # 飘动速度
@export var direction: Vector2 = Vector2.RIGHT  # 飘动方向
@export var lifetime: float = 8.0          # 在屏幕上存在的时间

# NonStopDebateConfig — 一场无休止议论的配置
class_name NonStopDebateConfig extends Resource
@export var phrases: Array[DebatePhrase]          # 所有发言
@export var spawn_interval: float = 1.5           # 发言生成间隔
@export var noise_phrases: Array[String]          # 杂音文本池
```

### 2.7 反论/反驳战（Rebuttal Showdown）

> **出现时机**：当主角提出某个推理后，某一角色强烈反对，进入 1v1 的激烈交锋。

#### 2.7.1 核心玩法

1. 对手连续发出 "反驳语句"，从屏幕中央向主角方向推进
2. 玩家需要在精确时机 **按键/滑动** 来 "斩断" 这些语句
3. 节奏越来越快，失误扣精神力
4. 所有语句斩断后，主角给出决定性反击

#### 2.7.2 实现思路

**最简单的实现**：类似音游的按键判定

```
屏幕上方出现对手台词 → 台词从左到右移动 → 到达判定线时按下按键 → 成功斩断
                                                        ↓ 按错/没按
                                                    精神力 -10
```

### 2.8 拼字/拼词（Hangman's Gambit / Bullet Time Battle / Kanji Assembly）

> **出现时机**：当需要玩家自己 **推理出某个关键词**（人名、凶器、作案手法等）时触发。

#### 2.8.1 核心玩法概述

拼字是弹丸论破中 **最灵活多变** 的小游戏类型。根据目标答案的样式不同，分为三种模式：

| 模式 | 目标答案 | 可选素材 | 玩法 | 系列出处 |
|------|---------|----------|------|---------|
| **拼字母** | 英文单词（如 `HOPE`、`DESPAIR`） | 英文字母 A-Z（含干扰项） | 旋转环选字母，按顺序拼词 | 弹丸1/2 |
| **拼汉字** | 单个汉字（如 `殺`、`密室`） | 偏旁部首、汉字组件 | 多个碎片飞过，按顺序击打合成为字 | 弹丸2/V3 |
| **拼词汇** | 汉语/日语词汇（如 `密室杀人`、`完美犯罪`） | 完整单个汉字/假名 | 从飘动的字中按顺序选出，组成短语 | 弹丸V3 |

> **设计原则**：三种模式共用同一套底层框架（计时→选字→判定→扣分），仅输入素材和UI布局不同。配置文件中指定模式类型即可切换。

---

#### 2.8.2 模式一：拼字母（Letter Hangman）—— 旋转环选字母

**玩法描述**：
- 目标是一个英文单词（如 `HOPE`），显示为 `_ _ _ _` 空槽
- 若干字母（含干扰项）排列在圆形轨道上旋转
- 玩家按确认键选取当前高亮的字母
- 选对→填入下一个空槽；选错→字母弹飞，扣精神力

**视觉风格**：
- 中央显示 "目标单词空槽" + 提示文字
- 外围一圈字母围绕中心旋转（类似俄罗斯轮盘）
- 高亮的字母放大、发光、带瞄准框
- 被选中的字母飞入空槽（成功）或弹飞（失败）

**配置数据结构**：


**Godot4 实现思路**：

```gdscript
# HangmanUI.gd — 字母环模式
extends CanvasLayer

@export var config: HangmanLetterConfig

var current_letter_index: int = 0     # 当前要填充的是第几个字母
var filled_slots: Array = []          # 已填充的字母
var time_left: float
var ring_angle: float = 0.0           # 当前旋转角度
var highlighted_index: int = 0        # 当前高亮的字母在环中的索引

func _ready():
    filled_slots.resize(config.target_word.length())
    filled_slots.fill("_")
    time_left = config.time_limit
    _setup_letter_ring()
    _update_display()

func _setup_letter_ring():
    var angle_step = TAU / config.available_letters.size()
    for i in range(config.available_letters.size()):
        var letter_node = letter_scene.instantiate()
        letter_node.text = config.available_letters[i]
        letter_node.ring_index = i
        $RingContainer.add_child(letter_node)
    _update_letter_positions()

func _process(delta):
    # 旋转环
    ring_angle += deg_to_rad(config.rotation_speed) * config.ring_direction * delta
    _update_letter_positions()
    # 确定当前高亮字母（最接近12点钟方向的）
    highlighted_index = _find_topmost_letter()
    _highlight_letter(highlighted_index)
    # 计时
    time_left -= delta
    if time_left <= 0:
        _on_time_up()

func _find_topmost_letter() -> int:
    # 找到Y坐标最小（最接近屏幕上方）的字母 = 位于12点方向
    var top_idx = 0
    var min_y = INF
    for i in range($RingContainer.get_child_count()):
        var pos = $RingContainer.get_child(i).position
        if pos.y < min_y:
            min_y = pos.y
            top_idx = i
    return top_idx

func _update_letter_positions():
    var angle_step = TAU / config.available_letters.size()
    for i in range($RingContainer.get_child_count()):
        var angle = i * angle_step + ring_angle
        var node = $RingContainer.get_child(i)
        node.position = Vector2(cos(angle), sin(angle)) * config.ring_radius

func _input(event):
    if event.is_action_pressed("confirm"):
        var selected_letter = config.available_letters[highlighted_index]
        if selected_letter == config.target_word[current_letter_index]:
            # 正确！字母飞入空槽
            filled_slots[current_letter_index] = selected_letter
            current_letter_index += 1
            _animate_letter_fly_in(highlighted_index)
            _update_display()
            if current_letter_index >= config.target_word.length():
                _on_word_complete()
        else:
            # 错误！字母弹飞
            _animate_letter_bounce_off(highlighted_index)
            DebateManager.damage_hp(5)

func _animate_letter_fly_in(ring_index: int):
    # 字母从环中位置飞到空槽位置
    var letter = $RingContainer.get_child(ring_index)
    var target_pos = $WordDisplay.get_slot_position(current_letter_index - 1)
    var tween = create_tween()
    tween.tween_property(letter, "position", target_pos, 0.3)
    tween.parallel().tween_property(letter, "modulate:a", 0.0, 0.3)
    # 播放音效
    AudioManager.play("hangman_correct")

func _animate_letter_bounce_off(ring_index: int):
    var letter = $RingContainer.get_child(ring_index)
    letter.modulate = Color.RED
    var tween = create_tween()
    tween.tween_property(letter, "position", letter.position * 1.3, 0.15)
    tween.tween_property(letter, "position", letter.position / 1.3, 0.15)
    tween.tween_property(letter, "modulate", Color.WHITE, 0.1)
    AudioManager.play("hangman_wrong")

func _update_display():
    $WordDisplay.text = " ".join(filled_slots)
```

#### 2.8.3 模式二：拼汉字（Kanji Assembly）—— 汉字组件合成

**玩法描述**：
- 目标是一个 **汉字**（如 `殺`、`鍵`、`謎`）
- 该汉字的 **偏旁部首或笔画组件**（如 `杀` + `殳` = `殺`）作为独立碎片从屏幕各方向飞过
- 玩家需 **按正确顺序** 击打/收集碎片，合成完整汉字
- 例如拼 `殺`：碎片① `杀` → 碎片② `殳` → 按顺序收集 → 合成为 `殺`

**关键与拼字母的区别**：
- 素材不是"单个字母"而是 "汉字组件/偏旁部首"
- 碎片数量少（通常 2-3 个），但移动速度快、方向复杂
- 强调 "顺序"：先击打左边偏旁，再击打右边组件
- 视觉上组件拼合的动画更有满足感

**配置数据结构**：

```gdscript
# HangmanKanjiConfig.gd
class_name HangmanKanjiConfig extends Resource
@export var mode: String = "kanji"
@export var target_kanji: String               # "殺"
@export var components_in_order: Array[String]  # ["杀", "殳"] — 必须按此顺序收集
@export var distractor_components: Array[String] # 干扰组件 ["又", "刀", "木"]
@export var hint_text: String                   # "凶手使用的___人手法"
@export var time_limit: float = 25.0
@export var component_speed_min: float = 80.0
@export var component_speed_max: float = 200.0
```

**Godot4 实现思路**：

```gdscript

    for comp in config.distractor_components:

        frag.is_correct = item["is_correct"]

func _animate_fragment_to_target(frag, order_index: int):

    tween.tween_property(frag, "position", $TargetArea.position + target_offset, 0.3)
    tween.parallel().tween_property(frag, "scale", Vector2(1.5, 1.5), 0.3)
```

#### 2.8.4 模式三：拼词汇（Word/Phrase Assembly）—— 多汉字顺序排列

**玩法描述**：
- 目标是多个汉字组成的 **词语/短语**（如 `密室杀人`、`完美犯罪`、`不在场证明`）
- 每个字符（单字或二字单元）作为独立选项在屏幕上出现
- 玩家需要 **按词汇的正确顺序**，依次选择/点击字符
- 选错顺序或选了无关字符 → 扣精神力

**典型场景示例**：
- 提示："真凶制造了怎样的犯罪？" → 空槽 `____ ____`
- 飘动选项：`密` `室` `杀` `人` `自` `意` `外`
- 玩家依次点击 `密` → `室` → `杀` → `人` → 拼出 `密室杀人` ✓

**与模式一/二的区别**：
- 素材本身是 **完整的字**（不难辨认），难度在于 **选出正确的字并按正确顺序排列**
- 没有"旋转环"，而是多个字在有限区域内飘动/脉动
- 类似 "词句拼图" 的手感

**配置数据结构**：

```gdscript

        _animate_widget_reject(widget)
        DebateManager.damage_hp(5)

func _animate_widget_to_slot(widget, slot_index: int):
    var target_pos = $SlotContainer.get_child(slot_index).global_position
    var tween = create_tween()
    tween.tween_property(widget, "global_position", target_pos, 0.3)
    tween.parallel().tween_property(widget, "scale", Vector2(1.3, 1.3), 0.3)
    widget.set_disabled(true)
    AudioManager.play("word_piece_correct")

func _update_slots_display():
    for i in range(config.target_chars.size()):
        var slot_label = $SlotContainer.get_child(i)
        if i < filled_chars.size():
            slot_label.text = filled_chars[i]
            slot_label.modulate = Color.WHITE
        else:
            slot_label.text = "＿"
            slot_label.modulate = Color.GRAY
```

#### 2.8.5 三种模式对比总结

| 维度 | 拼字母 | 拼汉字 | 拼词汇 |
|------|--------|--------|--------|
| **素材粒度** | 单个字母 (A-Z) | 偏旁部首/汉字组件 | 完整汉字 |
| **素材数量** | 多（8-15个） | 少（4-8个） | 中等（6-12个） |
| **素材运动方式** | 环绕中心旋转 | 从屏幕边缘飞入 | 随机飘动或网格排列 |
| **玩家操作** | 按确认键 = 取环上当前字母 | 点击飞过的碎片 | 点击飘动的字 |
| **核心难度** | 在干扰字母中快速辨认 | 辨认组件+记住顺序 | 从干扰字中选出正确组合 |
| **典型时间限制** | 20-30秒 | 15-25秒 | 20-30秒 |
| **拼错惩罚** | -5 HP | -5 HP | -5 HP |

### 2.9 高潮再现推理（Climax Inference）

> **出现时机**：学级裁判的 **最后一环**，所有证据齐备后，用漫画分格的形式回顾整个案件。

#### 2.9.1 核心玩法

1. 屏幕显示一条 "漫画分格条"（横向排列的空白格子）
2. 每个格子代表案件的一个阶段，需要填入对应的 "事件图块"
3. 屏幕下方散落着若干事件图块（文字+图片缩略图）
4. 玩家在 **时间限制内** 将正确的图块拖入正确的格子
5. 全部正确填入 → 真相大白 → 进入认罪/处刑表演

#### 2.9.2 数据结构

```gdscript
# ClimaxPanel — 一个漫画格子
class_name ClimaxPanel extends Resource
@export var title: String              # "被害者被引诱到图书馆"
@export var description: String        # 详细描述
@export var correct_tile_index: int    # 正确的事件图块索引
@export var illustration: Texture2D   # 漫画插图

# ClimaxInferenceConfig — 整场高潮再现配置
class_name ClimaxInferenceConfig extends Resource
@export var panels: Array[ClimaxPanel]     # 所有分格
@export var tiles: Array[ClimaxTile]       # 所有可拖入的图块（含干扰项）
@export var time_limit: float = 60.0
```

#### 2.9.3 实现思路

```gdscript
# ClimaxInferenceUI.gd
extends CanvasLayer

@export var config: ClimaxInferenceConfig
@export var panel_scene: PackedScene     # ClimaxPanelSlot.tscn
@export var tile_scene: PackedScene      # ClimaxTile.tscn

var panel_slots: Array = []
var tile_nodes: Array = []
var filled_count: int = 0

func _ready():
    _build_panel_slots()
    _build_tile_tray()
    _start_timer()

func _build_panel_slots():
    for i in range(config.panels.size()):
        var slot = panel_scene.instantiate()
        slot.panel_data = config.panels[i]
        slot.slot_index = i
        slot.tile_dropped.connect(_on_tile_dropped)
        $PanelRow.add_child(slot)
        panel_slots.append(slot)

func _build_tile_tray():
    # 随机排序图块，增加难度
    var shuffled = config.tiles.duplicate()
    shuffled.shuffle()
    for tile_data in shuffled:
        var tile = tile_scene.instantiate()
        tile.tile_data = tile_data
        tile.draggable = true
        $TileTray.add_child(tile)
        tile_nodes.append(tile)

func _on_tile_dropped(slot_index: int, tile_index: int):
    var slot = panel_slots[slot_index]
    if tile_index == slot.panel_data.correct_tile_index:
        # 正确！
        slot.set_filled(tile_nodes[tile_index])
        filled_count += 1
        if filled_count == config.panels.size():
            _on_all_complete()
    else:
        # 错误！图块弹回原位
        tile_nodes[tile_index].reset_position()
        DebateManager.damage_hp(5)
```

### 2.11 通用支撑系统（所有小游戏共享）

> 以下系统是学级裁判的"基础设施"，不属于某个小游戏独有，但贯穿所有小游戏。

#### 2.11.1 言弹子弹动画系统（Evidence Bullet Animation）

**玩法描述**：
- 无休止议论中，选中言弹后，一颗"子弹"从言弹环飞向目标文字
- 子弹带有拖尾特效
- 命中成功→爆炸特效+文字碎裂；命中失败→弹飞特效
- 这是一个独立的视觉组件，在反论中也可复用（"斩断"的刀光）

**Godot4 实现思路**：

```gdscript
# EvidenceBullet.gd
extends Node2D

signal hit_target(success: bool)

var start_pos: Vector2
var target_pos: Vector2
var evidence_id: String
var speed: float = 800.0

func _ready():
    position = start_pos

func _process(delta):
    position = position.move_toward(target_pos, speed * delta)
    if position.distance_to(target_pos) < 5.0:
        _on_arrive()

func _on_arrive():
    # 到达目标位置后，由 NonStopDebateUI 判断是否正确目标
    hit_target.emit(true)  # 实际判断在调用方
    queue_free()
```

#### 2.11.2 裁判中暂停菜单（Trial Pause Menu）

**玩法描述**：
- 裁判中按 ESC 暂停
- 暂停菜单提供：继续游戏、查看证据、查看规则、返回标题、系统设置
- 暂停时背景模糊/变暗
- 倒计时小游戏中暂停=隐藏倒计时

**Godot4 实现思路**：
```gdscript
# TrialPauseMenu.gd
extends CanvasLayer

func _input(event):
    if event.is_action_pressed("pause"):
        if visible:
            _resume()
        else:
            _pause()

func _pause():
    get_tree().paused = true
    visible = true
    # 添加模糊着色器到背景
    $Background.material = preload("res://shaders/blur.tres")

func _resume():
    get_tree().paused = false
    visible = false
```

#### 2.11.3 手柄/控制器支持（Gamepad Support）

所有小游戏必须同时支持键盘鼠标和手柄操作：

| 小游戏 | 键盘/鼠标操作 | 手柄操作 |
|--------|-------------|---------|
| 无休止议论 | 鼠标移动准星 / 左键发射 | 右摇杆移动准星 / RT 发射 |
| 反论 | 空格键斩击 | A 键斩击 |
| 拼字（字母环） | 空格键确认 | A 键确认 |
| 拼字（汉字/词汇） | 鼠标点击碎片 | 左摇杆移动光标 / A 键选择 |
| 高潮再现 | 鼠标拖拽图块 | 左摇杆移动 / A 抓起 / 放下 |

统一在 `Input Map` 中定义操作名称（如 `confirm`、`slash`、`fire_evidence` 等），代码中只使用 `Input.is_action_pressed("confirm")`，不检测具体按键。

#### 2.11.4 多语言/本地化支持（i18n）

虽然剧本JSON已支持 `locales/` 目录，但核心系统的本地化策略需要明确：

- **剧本本身不本地化**：每语言一个独立剧本目录（如 `story/zh_cn/`、`story/en_us/`）
- **UI 文字使用 CSV/JSON 键值表**：

- 在 UI 脚本中使用 `tr("ui.continue")` 调用 Godot 内置的翻译系统

#### 2.11.5 3D 角色立绘板（Billboard Characters）

在学级裁判的 3D 场景中，角色模型可以用两层方案：
1. **开发阶段**：`Sprite3D` 节点（始终面朝摄像机）+ 角色立绘贴图
2. **打磨阶段**：替换为完整 3D 模型

```gdscript
# BillboardCharacter.gd
extends Sprite3D

func _process(_delta):
    # 始终面朝摄像机
    var cam = get_viewport().get_camera_3d()
    if cam:
        look_at(cam.global_position, Vector3.UP)
```

#### 2.11.6 裁判节奏控制（Trial Pacing）

学级裁判的节奏感非常重要，需要在以下方面做节奏控制：

- **对话与小游戏的交替**：不要连续两场同样的小游戏，中间穿插剧情对话
- **BGM 情绪递进**：开场 BGM → 辩论 BGM → 高潮 BGM → 处刑 BGM
- **HP 压力曲线**：初期小游戏简单（扣 HP 少），越往后越难
- **摄像机变换节奏**：对话时多用特写，全员讨论时拉远，反击时快速切镜

这些由剧本编写者控制（通过指令编排），不是编程问题，但程序需保证每个指令的响应时间一致且流畅。

## 三、世界观与背景设定

### 3.1 主题：绝望 vs 希望

- **核心冲突**：黑白熊诱导学生自相残杀 → 产生绝望 → 主角团用推理揭示真相 → 重燃希望
- **美术风格**：
  - 整体偏暗色调（深紫、血红、黑色）
  - 裁判场背景用 "电子屏 + 哥特风格" 混搭
  - 角色立绘色彩鲜明，与灰暗背景形成对比（突出个体）
  - 关键场景大量使用聚光灯效果（学级裁判的 "舞台感"）

### 3.2 黑白熊角色设计

**职责**：
- 宣布游戏规则（"校规"）
- 每次事件后发布 "黑白熊档案"（尸检报告等）
- 主持学级裁判（嘲讽、挑拨）
- 执行处刑

**在裁判场中的表现**：
- 裁判开始时从天花板降落（或从中心台升起）
- 坐在中央最高处的 "御座" 上
- 时不时跳出来插话、嘲讽、给出提示
- 处刑时切换到专门的黑白熊 3D 动画场景

### 3.3 封闭空间视觉风格

**学校环境**：
- 铁窗封死的窗户
- 监视摄像头随处可见
- 走廊灯光昏暗，闪烁故障效果
- 墙壁上贴满黑白熊海报和校规
- 脱出区域（如大门、楼梯间）用铁栅栏 + 警示灯标记

**3D裁判场环境**：
- 环形站台 16 个，每个有角色的校徽
- 中心高台（黑白熊座位）
- 四周环绕巨大的屏幕/显示器（显示证据图片）
- 背景：黑暗虚空或抽象的审判空间
- 灯光：聚光灯只照亮当前发言者

1. **编剧友好**：剧本用 JSON/Tres 存储，非程序员可直接在文本编辑器中编写和修改

### 4.5 剧本解释器核心逻辑

```gdscript
# ScriptInterpreter.gd (Autoload)
extends Node

signal command_executed(cmd_id: String)
signal script_finished(script_id: String)

var current_script: Array = []
var current_index: int = 0
var is_waiting_input: bool = false
var is_auto_mode: bool = false

var command_handlers: Dictionary = {}  # 指令 → 处理函数的映射

func _ready():
    _register_handlers()

func _register_handlers():
    command_handlers["dialogue"] = _handle_dialogue
    command_handlers["set_bg"] = _handle_set_bg
    command_handlers["bgm"] = _handle_bgm
    command_handlers["sfx"] = _handle_sfx
    command_handlers["screen_effect"] = _handle_screen_effect
    command_handlers["show_character"] = _handle_show_character
    command_handlers["hide_character"] = _handle_hide_character
    command_handlers["load_scene"] = _handle_load_scene
    command_handlers["add_evidence"] = _handle_add_evidence
    command_handlers["choice"] = _handle_choice
    command_handlers["jump"] = _handle_jump
    command_handlers["label"] = _handle_label  # 实际上就是空操作，标记被jump使用
    command_handlers["if"] = _handle_if
    command_handlers["set_flag"] = _handle_set_flag
    command_handlers["affection"] = _handle_affection
    command_handlers["start_nonstop_debate"] = _handle_nonstop_debate
    command_handlers["start_rebuttal"] = _handle_rebuttal
    command_handlers["start_hangman"] = _handle_hangman
    command_handlers["start_climax_inference"] = _handle_climax_inference
    command_handlers["trial_result"] = _handle_trial_result
    command_handlers["save_prompt"] = _handle_save_prompt
    command_handlers["comment"] = _handle_comment  # 空操作
    command_handlers["skill_select"] = _handle_skill_select

func load_script(script_path: String):
    var file = FileAccess.open(script_path, FileAccess.READ)
    var json = JSON.parse_string(file.get_as_text())
    current_script = json["events"]
    current_index = 0
    _execute_next()

func _execute_next():
    while current_index < current_script.size():
        var cmd = current_script[current_index]
        var cmd_type = cmd.get("cmd", "")
        
        if command_handlers.has(cmd_type):
            var should_await = command_handlers[cmd_type].call(cmd)
            current_index += 1
            if should_await:
                # 等待用户输入或异步完成
                return
        else:
            push_warning("Unknown command: " + cmd_type)
            current_index += 1
    
    # 脚本执行完毕
    script_finished.emit(current_script.get("script_id", ""))

func _handle_dialogue(cmd: Dictionary) -> bool:
    # 返回 true 表示需要等待用户输入
    EventBus.emit_signal("show_dialogue", cmd)
    return true  # 等待点击

func _handle_nonstop_debate(cmd: Dictionary) -> bool:
    EventBus.emit_signal("start_nonstop_debate", cmd["config_id"])
    return true  # 等待小游戏完成

func next():
    # 由玩家点击/自动模式触发
    _execute_next()

func skip_to_label(label_name: String):
    for i in range(current_script.size()):
        var cmd = current_script[i]
        if cmd.get("cmd") == "label" and cmd.get("name") == label_name:
            current_index = i
            _execute_next()
            return
    push_error("Label not found: " + label_name)
```

### 4.6 剧本编辑器友好性设计

为了让非程序员能轻松编写剧本，建议提供：

4. **可视化编辑器（远期目标）**：在 Godot 编辑器内通过插件提供剧本可视化编辑

### 5.0 OpenCode 执行计划的组织原则

1. **每步只做一件事**：每个步骤对应一个独立的、可验证的任务
2. **自底向上**：先搭建基础设施（資源管理、场景管理），再实现具体功能
3. **先2D后3D**：先完成2D日常/非日常框架，再攻克3D学级裁判
4. **每步产出可见**：每完成一步，都能在 Godot 编辑器中看到可运行的阶段性成果
5. **遵循已有构建说明**：基于本手册的架构设计逐步实现

### C.2 架构实现要点

#### C.2.1 Autoload 加载顺序

当前项目 `project.godot` 中注册的 Autoload 及加载顺序（顺序重要）：

```
EventBus → SaveLoadManager → SceneManager → AudioManager
  → GameManager → ScriptInterpreter → EvidenceManager
  → CharacterPortraitManager → MainGame
```

MainGame 必须在最后，因为它依赖其他所有 Autoload 的初始化结果。

#### C.2.2 主场景策略

- 原计划使用 `.tscn` 场景文件 + `ext_resource` 引用脚本
- 实际采用 **MainGame Autoload** 方案：`scenes/main.tscn` 为空 Node，由 `MainGame` Autoload 的 `_ready()` 动态创建整个 2D 世界和 UI 层
- 原因：手写 `.tscn` 的 `ext_resource` UID 格式兼容性不确定，Autoload 方案更可靠

#### C.2.3 输入按键设计变更（与原始手册的差异）

原手册规划的按键分工在实现时暴露了冲突问题，经过多轮迭代最终方案：

| 操作 | 原手册规划 | 最终实现 | 原因 |
|------|-----------|---------|------|
| 对话互动 | confirm（Enter/Space） | **interact（E）** | Enter 同时用于推进对话，会导致连按冲突 |
| 推进对话 | 鼠标点击 | confirm + 鼠标左键 | 增加 Enter/Space 支持，更符合视觉小说习惯 |
| 发射言弹 | 鼠标左键 | 鼠标左键 | 保持不变 |
| 暂停 | Esc | Esc | 保持不变 |

**新增 Input Map 动作 `interact`**：映射 E 键（keycode=69）+ 手柄 A 键，用于 NPC 对话和调查点交互。

#### C.2.4 碰撞系统

- **NPC**：`InteractableNPC` 内部自动创建 `StaticBody2D` + `CircleShape2D`（半径 24），与 `Area2D` 检测范围（半径 60）分离。玩家在 ~60px 处看到提示，在 ~40px 处被阻挡
- **调查点**：`InvestigationPoint` 新增 `@export var has_collision: bool = true` 字段，`true` 时自动创建 `StaticBody2D` + `RectangleShape2D`（40×40），`false` 时不阻挡玩家
- **玩家**：`PlayerController` 创建 `CircleShape2D`（半径 16），通过 `move_and_slide()` 处理物理碰撞

#### C.2.5 剧本-交互互斥

`InteractableNPC` 和 `InvestigationPoint` 的 `_input()` 中均增加了守卫：

```gdscript
if ScriptInterpreter.is_executing or ScriptInterpreter.is_waiting_input:
    return
```

防止在对话/选项/小游戏进行中反复触发 NPC 交互。

#### C.2.6 逐字显示 vs 键盘推进

`DialogueBox._input()` 在可见时同时响应鼠标左键和 confirm（Enter/Space）动作：

```
点击/按键 → 如果文字未显示完 → 立即显示全部文字
         → 如果文字已显示完 → 关闭对话框 → 触发 EventBus.dialogue_next → 进入下一句
```

### D.7 无休止议论相关设计决策

#### 言弹交互演变

原手册规划两种模式（环形菜单/列表选择）。实际实现经历了三次迭代：
1. 环形菜单 — Tab 打开，鼠标点击（已废弃）
2. Tab 循环切换 — 无列表（已废弃）
3. **最终方案**：点按 Tab（<0.3s）切换言弹，长按 Tab 弹出列表 + 滚轮选择 + 松开确认

原因：环形菜单与漂浮文字的点击冲突，且需要大量按钮管理；纯循环切换缺乏可视列表；最终方案兼顾了便捷切换和可视选择。

#### 矛盾点从整句到句中词

原手册将整条发言标记为矛盾。实际实现改为 `hotspot` 模式：
- 一句话可包含多个矛盾词（真/假）
- 真矛盾（金色）+ 假矛盾（橙色）+ 普通文字（灰色）
- `find_hotspot_at()` 使用局部坐标系检测点击命中
- 每个 hotspot 可配置 `fail_dialogue`（击错后的提示对话）

#### 发言循环

原手册一次生成全部发言。实际实现为轮流发言：
- 一次显示一条发言，`speak_duration` 控制显示时长（2.5-4.5 秒）
- 全部发言轮完 → 主角总结思考（复用裁判场对话框）→ 重新开始循环
- 只有击破真矛盾才结束议论

#### 摄像机运镜配置

议论中摄像机支持在剧本中配置：

**debate JSON 配置字段**：
```json
{
  "camera_transition": "rotate",  // cut/rotate/sweep/zoom
  "camera_lock": "drift"          // steady/drift/breathe/dolly
}
```

**过渡预设**：`rotate`（绕中心旋转）、`cut`（硬切）、`sweep`（拉远推近）、`zoom`（退远推进）

**锁定预设**：`steady`（静止）、`drift`（微漂移）、`breathe`（缩放呼吸）、`dolly`（横向滑轨）

#### 发言 UI 布局

- 每条发言为独立 `FloatingPhrase` Control，含发言人标签 + 多段文字 Label
- 普通文字灰色、假矛盾橙色、真矛盾金色
- 准星跟随鼠标，`_process` 中实时检测悬停并改变准星颜色
- `clampf` 硬约束确保不飘出屏幕
- 发言者切换触发摄像机旋转运镜 + 站台高亮

| `PanoramaSkyMaterial` 属性名不稳定 | 改用 `BG_COLOR` 纯色背景 |
| `Dictionary.get()` 返回 Variant | 禁止 `:=` 推断，全部显式声明类型 |

| `Camera3D.current_enabled` | 改为 Godot 4 的 `enabled` |
| `RichTextLabel.fit_content` 默认宽为 0 | 预设 `size = Vector2(900, 60)` 再渲染 |
